Aprender React leyendo tu propio proyecto

Guía sobre PlayaDetalle.tsx pensada para un programador que viene de Java.

1 El método para leer cualquier app React

No leas los archivos por orden alfabético ni por tamaño. Lee siguiendo el flujo de datos, que siempre es el mismo:

index.tsx→ App.tsx (router)→ Página→ Fetch / estado→ Render (JSX)→ Componentes hijos

Ante cualquier proyecto desconocido: abres src/index.tsx, luego App.tsx, y desde ahí "tiras del hilo". Ese es el 80% del truco.

Regla de oro: un componente es una función que se re-ejecuta entera cada vez que su estado o sus props cambian. No hay mutación in-place como en un objeto Java vivo; se vuelve a llamar al método y React compara el resultado nuevo con el anterior.

4 La línea clave, troceada — línea 796

const PlayaDetallePage: React.FC = () => {

Léela como una frase: "declaro una constante llamada PlayaDetallePage, de tipo componente-de-función, cuyo valor es una función que no recibe nada y devuelve JSX."

TrozoQué esSímil Java

PlayaDetallePageEl nombre. Empieza en mayúscula a propósitoNombre de la clase/método

En Java sería, aproximadamente:

final Supplier<View> PlayaDetallePage = () -> { /* … */ return vista; };

Tres cosas que sorprenden viniendo de Java

1. Un componente es una variable, no una clase. En React moderno una "pantalla" es literalmente una función guardada en una const. No hay class ... extends Component. La función es el componente.
2. La mayúscula es obligatoria y semántica. React usa la inicial mayúscula para distinguir tus componentes (<PlayaDetallePage/>) de las etiquetas HTML nativas (<div/>). Si la llamaras playaDetallePage, React creería que es una etiqueta HTML desconocida y no la pintaría. Es una convención con efecto funcional.
3. React.FC es solo el tipo. FC = Function Component. Es opcional (podrías omitir : React.FC), pero documenta la intención. Como escribir Runnable r = … en vez de var r = ….

Y el cierre — línea 958

export default PlayaDetallePage;

export default = "esta es la pieza pública principal del archivo". Como marcar public la clase que otros importan. Es lo que permite que App.tsx haga import PlayaDetallePage from './pages/PlayaDetalle'.

5 Paso a paso del componente principal

Empieza a leer aquí (línea 796). Es el único con lógica de verdad; el resto solo pinta.

Paso 1 — Leer el input de la URL 797–799

const { codigo } = useParams<{ codigo: string }>();
const history = useHistory();

useParams saca el :codigo de la ruta /playas/:codigo. Como un @PathVariable de Spring. history sirve para navegar (goBack, push) — como un response.sendRedirect.

Paso 2 — Declarar el estado 800–801

const [datos, setDatos] = useState<PlayaDetalleData | null>(null);
const [error, setError] = useState(false);

Dos campos: los datos de la playa (empiezan null) y un flag de error. setDatos/setError son sus setters. Cada set* dispara un re-render. La sintaxis [valor, setValor] es desestructuración de array: useState devuelve un par y le pones nombre a cada posición.

Paso 3 — Cargar datos al montar 803–807 · el trozo más importante

useEffect(() => {
  getDetallePlaya(codigo)
    .then(setDatos)
    .catch(() => setError(true));
}, [codigo]);

Traducción: "cuando el componente aparezca (o cuando codigo cambie), lanza la llamada HTTP asíncrona; cuando responda, guarda el resultado en datos; si falla, marca error". El [codigo] es la dependencia: solo re-ejecuta si codigo cambia. .then(setDatos) equivale a un CompletableFuture.thenAccept(this::setDatos).

El ciclo completo: monta → datos es null → pinta el spinner → llega el fetch → setDatos → se vuelve a ejecutar toda la función → ahora datos tiene valor → pinta las cards.

Paso 4 — Cálculos derivados 809–812

const [selectedDay, setSelectedDay] = useState(0);
const pred = datos?.prediccionCompleta;
const safeDayIndex = pred ? Math.min(selectedDay, pred.dias.length - 1) : 0;

selectedDay = qué pestaña de día está activa. pred y safeDayIndex no son estado: se recalculan en cada render a partir del estado, como variables locales dentro de render(). El ?. es null-safe (igual que el ?. de Kotlin o un Optional encadenado).

6 El return: JSX condicional y bucles

El return es el "Thymeleaf/JSP" del método. Tres sintaxis que confunden al venir de Java:

a) {condición && <Componente/>} — if sin else ej. 828

{error && (
  <div className="error-container">
    <p>{t('detalle.errorCarga')}</p>
  </div>
)}

"Si error es true, pinta esto; si no, nada."

b) {cond ? <A/> : <B/>} — ternario = if/else ej. 869

{pred && pred.dias.length > 0 ? (
  /* hay predicción completa: DaySelector + ForecastHero + ... */
) : (
  /* fallback: WeatherCard legacy */
)}

c) {lista.map(x => <Item/>)} — bucle for ej. 872 y 188

{fechas.map((fecha, i) => (
  <button key={fecha} onClick={() => onSelect(i)}>
    {dayTitle(fecha, t, idioma)}
  </button>
))}

Un bucle que produce N elementos, como un stream().map().collect() que devuelve vistas. El key es obligatorio: le da a React una identidad estable para cada elemento de la lista (para repintar solo lo que cambia).

Los tres estados de pantalla se pintan por orden de guarda: {error && …} (828) → {!datos && !error && …} spinner (834) → {datos && (…)} contenido real (841). Solo uno se cumple a la vez. Es el patrón loading / error / success que verás en TODA app con fetch.

7 Datos bajan por props, eventos suben por callbacks

Dentro del {datos && …} se ensamblan los sub-componentes pasándoles datos por props, como llamar métodos:

<FlagBanner cruzRoja={datos.cruzRoja} />
<ForecastHero dia={pred.dias[safeDayIndex]} ... />
<DaySelector ... onSelect={setSelectedDay} />

Fíjate en onSelect={setSelectedDay}: le pasa el setter al hijo como callback. El flujo:

usuario pulsa un día→ hijo llama onSelect(i)→ setSelectedDay(i) en el padre→ re-render→ cambia safeDayIndex→ se repinta el día elegido

En Java sería pasarle un Consumer<Integer> al hijo. Los datos bajan; los eventos suben.

¿Dónde vive el estado?

DaySelector (180–200) no tiene estado propio: recibe selectedDay y avisa hacia arriba con onSelect. Es un componente controlado ("tonto"). En cambio WeatherCard (518) y CruzRojaCard (712) sí tienen estado propio:

const [expanded, setExpanded] = useState(defaultExpanded);

...porque "estoy plegado/desplegado" es asunto interno de la card; al padre no le importa.

Regla práctica: el estado vive en el nivel más bajo que lo necesite. Si dos hermanos lo comparten, súbelo al padre común (eso se llama lifting state up).

8 Por qué se lee del main hacia arriba

Si intentas leer el archivo de arriba abajo (línea 1 → 958), te comes 792 líneas de piezas sueltas sin saber para qué sirven, y te rindes.

En su lugar: salta al main (796) primero. Ves el esqueleto — "esta pantalla es un FlagBanner + un ForecastHero + unas cards" — y solo cuando te preguntes "¿qué pinta exactamente ForecastHero?", subes a leer esa pieza (272). Lees las piezas bajo demanda, no en orden de archivo. Es el mismo método "sigue el dato": entras por el main y profundizas solo cuando el flujo te lleva ahí.

9 Olores de código (tu ojo crítico)

God Component. Este archivo es funcional pero tiene 958 líneas y 13 componentes en un solo fichero. En el trabajo lo verás marcado como deuda técnica. La refactorización natural: mover cada sub-componente a components/detalle/FlagBanner.tsx, etc. La lógica (pasos 1–4) está limpia; el problema es solo que todo convive en un archivo.
Lo que este proyecto NO te enseña (y verás en el trabajo): estado global con librería (Redux/Zustand), data fetching moderno (React Query/SWR) en lugar de fetch en useEffect, y Next.js / Server Components. Entender la versión "manual" primero hace que esas herramientas tengan sentido después.

10 Tu ejercicio

🎯 Analiza FlagBanner (línea 102)

Aplica las 5 preguntas a cualquier componente:

  1. ¿Qué props recibe? (mira la firma React.FC<{…}>)
  2. ¿Qué estado tiene? (busca los useState)
  3. ¿Qué efectos hace? (busca los useEffect)
  4. ¿Qué renderiza? (baja al return)
  5. ¿A qué hijos llama? (los <ComponentesConMayúscula/>)
const FlagBanner: React.FC<{ cruzRoja?: PlayaDetalleData['cruzRoja'] }> = ({ cruzRoja }) => {
  const { t } = useIdioma();
  const estado = estadoBandera(cruzRoja);
  if (estado === 'sinDatos') {
    return null;   // no pinta nada
  }
  const colorClass = flagColorClass(cruzRoja!.bandera);
  return (
    <div className="flag-banner">
      /* ... indicador + info de la bandera ... */
    </div>
  );
};

Pistas de respuesta: (1) una prop opcional cruzRoja; (2) ningún useState → no tiene estado; (3) ningún useEffect; (4) devuelve un <div>... o null si no hay datos (un componente puede decidir no pintar nada); (5) no llama a otros componentes tuyos, solo HTML. Es el componente más simple del archivo: puro props → JSX.
